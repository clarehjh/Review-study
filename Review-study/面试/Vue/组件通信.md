# Vue 组件通信 （10 + 种方式）

## 一、Vue 组件通信全场景梳理（10 + 种方式）

### 1. 基础通信方式（通用）

表格


| 通信方式                  | 适用场景               | 核心原理 / 用法                                                                               | 优缺点                                                               |
| ------------------------- | ---------------------- | --------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| **Props / \$emit**        | 父子组件通信（最常用） | 父传子：子组件`defineProps`接收，父组件通过属性传递；子传父：子组件`emit`触发事件，父组件监听 | 优点：符合单向数据流，简单直观；缺点：仅支持父子，多层嵌套需层层传递 |
| **自定义 v-model**        | 父子组件双向绑定       | 父组件：`v-model:属性名="值"`；子组件：`defineProps`接收属性 +`emit('update:属性名', 新值)`   | 优点：简化双向绑定代码，符合 Vue 规范；缺点：仅支持父子              |
| **ref / \$refs**          | 父组件获取子组件实例   | 父组件给子组件加`ref="xxx"`，通过`ref.xxx`访问子组件的属性 / 方法                             | 优点：直接操作子组件，灵活；缺点：破坏封装，耦合度高                 |
| **\$parent / \$children** | 父子组件互访           | 子组件通过`$parent`访问父组件，父组件通过`$children`访问子组件（Vue2 为主）                   | 优点：简单；缺点：依赖组件层级，灵活性差，Vue3 中已弱化              |

### 2. 跨层级 / 任意组件通信

表格


| 通信方式                  | 适用场景                   | 核心原理 / 用法                                                                                                                     | 优缺点                                                                         |
| ------------------------- | -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **provide / inject**      | 隔代 / 跨层级组件通信      | 父组件：`provide('key', 值)`提供数据；后代组件：`inject('key')`注入数据                                                             | 优点：无需层层传递，适配深层嵌套；缺点：数据来源不清晰，建议配合 readonly 使用 |
| **mitt / 事件总线**       | 任意组件通信（无层级限制） | 1. 安装 mitt：`npm i mitt`；2. 创建总线：`const bus = mitt()`；3. 发送：`bus.emit('event', 数据)`；4. 接收：`bus.on('event', 回调)` | 优点：轻量、灵活，无层级限制；缺点：全局事件易混乱，需手动销毁监听             |
| **Vuex / Pinia**          | 大型项目全局状态管理       | Vuex：核心（State/Mutations/Actions/Getters/Modules）；Pinia：简化版，无 Mutations，支持 TS                                         | 优点：统一管理全局状态，支持模块化；缺点：小型项目接入成本高                   |
| **\$attrs / \$listeners** | 隔代组件透传               | `$attrs`：接收父组件传递的非 props 属性；`$listeners`：接收父组件监听的事件（Vue2 为主）                                            | 优点：透传属性 / 事件，无需手动转发；缺点：Vue3 中`$listeners`已合并到`$attrs` |

### 3. 特殊场景通信

表格


| 通信方式             | 适用场景                 | 核心原理 / 用法                                                                 | 优缺点                                                       |
| -------------------- | ------------------------ | ------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| **插槽（Slot）**     | 父子组件结构通信         | 父组件通过插槽传递 DOM / 组件，子组件通过`<slot>`接收                           | 优点：灵活传递结构，支持作用域插槽传数据；缺点：仅支持父子   |
| **全局变量 /window** | 任意组件通信（临时场景） | 挂载数据到`window`或全局对象，任意组件访问                                      | 优点：最简单；缺点：易造成全局污染，维护性差                 |
| **路由传参**         | 页面级组件通信           | 动态路由：`/path/:id`+`$route.params.id`；query：`/path?id=1`+`$route.query.id` | 优点：适配页面跳转传参；缺点：仅支持页面组件，参数暴露在 URL |

## 二、重点通信方式深度拆解（面试高频）

### 1. provide /inject（核心考点）

#### 核心用法

vue

```
<!-- 父组件：提供数据 -->
<script setup>
import { provide, reactive, readonly } from 'vue'
// 提供普通数据
provide('title', 'Vue 组件通信')
// 提供响应式数据（推荐 readonly 防止后代修改）
const user = reactive({ name: '张三', age: 20 })
provide('user', readonly(user)) // 只读，后代无法直接修改
// 提供方法（让后代通过方法修改数据）
provide('updateName', (newName) => {
  user.name = newName
})
</script>

<!-- 后代组件：注入数据 -->
<script setup>
import { inject } from 'vue'
const title = inject('title', '默认标题') // 第二个参数为默认值
const user = inject('user')
const updateName = inject('updateName')

// 修改数据（通过父组件提供的方法）
const changeName = () => {
  updateName('李四')
}
</script>
```

#### 面试核心考点

* 为什么要用 `readonly` 包装？防止后代组件直接修改父组件提供的响应式数据，破坏单向数据流；
* 数据是否响应式？是，父组件修改数据，后代组件会同步更新；
* 适用场景：组件库开发（如 Element Plus 组件嵌套）、深层嵌套的业务组件（如表单套表单）。

### 2. 自定义 v-model（高频手写题）

#### 基础用法（单个 v-model）

vue

```
<!-- 子组件 MyInput.vue -->
<template>
  <input 
    :value="modelValue" 
    @input="$emit('update:modelValue', $event.target.value)"
  />
</template>
<script setup>
// 接收默认的 modelValue 属性
defineProps(['modelValue'])
// 声明触发的事件
defineEmits(['update:modelValue'])
</script>

<!-- 父组件使用 -->
<template>
  <MyInput v-model="inputValue" />
</template>
<script setup>
import { ref } from 'vue'
import MyInput from './MyInput.vue'
const inputValue = ref('')
</script>
```

#### 多个 v-model（面试高频）

vue

```
<!-- 子组件 UserForm.vue -->
<template>
  <input :value="name" @input="$emit('update:name', $event.target.value)" />
  <input :value="age" @input="$emit('update:age', $event.target.value)" />
</template>
<script setup>
defineProps(['name', 'age'])
defineEmits(['update:name', 'update:age'])
</script>

<!-- 父组件使用 -->
<template>
  <UserForm v-model:name="userName" v-model:age="userAge" />
</template>
<script setup>
import { ref } from 'vue'
import UserForm from './UserForm.vue'
const userName = ref('张三')
const userAge = ref(20)
</script>
```

### 3. mitt / 事件总线（核心考点）

#### 完整使用示例（含销毁监听）

javascript

运行

```
// 1. 创建全局总线文件：src/utils/bus.js
import mitt from 'mitt'
const bus = mitt()
export default bus

// 2. 发送数据的组件 A
<script setup>
import bus from '@/utils/bus'
const sendData = () => {
  // 触发事件并传递数据
  bus.emit('user-change', { name: '李四', age: 21 })
}
</script>

// 3. 接收数据的组件 B
<script setup>
import { onUnmounted } from 'vue'
import bus from '@/utils/bus'

// 监听事件
const handleUserChange = (data) => {
  console.log('接收的数据：', data)
}
bus.on('user-change', handleUserChange)

// 组件卸载时销毁监听（必做，防止内存泄漏）
onUnmounted(() => {
  bus.off('user-change', handleUserChange)
})
</script>
```

#### 面试核心考点

* 为什么要销毁监听？防止组件卸载后仍监听事件，导致内存泄漏；
* mitt 对比 Vue2 原生事件总线（`new Vue()`）：mitt 更轻量（200B）、支持 TS、支持批量移除事件；
* 适用场景：中小型项目的跨组件通信，大型项目建议用 Pinia。

## 三、15k+ 前端高频面试题 & 标准答案

### 基础必问（80% 面试官会问）

#### 题目 1：Vue 中有哪些组件通信方式？分别适用于什么场景？

**标准答案**：

Vue 组件通信方式可按「组件层级」分为三类，核心场景如下：

1. **父子组件通信**：
   * Props/\$emit（最常用）：基础父子传值，符合单向数据流；
   * 自定义 v-model：简化父子双向绑定，如表单组件；
   * ref/\$refs：父组件需直接操作子组件实例（如调用子组件方法）；
2. **跨层级 / 隔代组件通信**：
   * provide/inject：深层嵌套组件（如组件库、多层表单），无需层层传递；
   * attrs/listeners：透传属性 / 事件，减少手动转发；
3. **任意组件通信**：
   * mitt / 事件总线：中小型项目，无层级限制，轻量灵活；
   * Vuex/Pinia：大型项目全局状态管理，支持模块化和状态追踪；
4. **特殊场景**：
   * 插槽：父子组件结构通信；
   * 路由传参：页面级组件跳转传参；
   * 全局变量：临时 / 简单场景（不推荐）。

#### 题目 2：Props 和 \$emit 的原理？如何实现父子组件双向绑定？

**标准答案**：

1. **Props/\$emit 原理**：
   * Props：父组件向子组件传递数据，子组件通过 `defineProps` 声明接收，Vue 会校验 props 类型 / 默认值，且遵循「单向数据流」（子组件不能直接修改 props）；
   * \$emit：子组件通过触发自定义事件，将数据传递给父组件，父组件监听事件并修改自身数据，间接实现子传父。
2. **父子双向绑定实现方式**：
   * 方式 1（基础）：父组件传递 props，子组件 `emit` 事件通知父组件修改；
   * 方式 2（推荐）：自定义 v-model，父组件用 `v-model:属性名="值"`，子组件接收 `属性名` 并触发 `update:属性名` 事件，语法更简洁，符合 Vue 规范。

#### 题目 3：provide/inject 的使用场景？为什么要配合 readonly 使用？

**标准答案**：

1. **使用场景**：
   适用于**隔代 / 深层嵌套组件**通信（如组件库开发、多层表单组件、布局组件嵌套），无需像 props 一样层层传递数据，简化代码。
2. **配合 readonly 的原因**：
   provide/inject 会传递响应式数据的引用，后代组件若直接修改该数据，会破坏「单向数据流」（数据来源不清晰，难以追踪修改）；
   用 `readonly` 包装提供的数据，可禁止后代组件直接修改，若需修改，可让父组件提供修改方法（如 `provide('updateFn', () => { ... })`），保证数据修改逻辑统一在父组件。

### 进阶高频（15k+ 重点）

#### 题目 4：mitt 事件总线的使用步骤？如何避免内存泄漏？

**标准答案**：

1. **mitt 使用步骤**：
   * 安装：`npm i mitt`；
   * 创建全局总线：`const bus = mitt()` 并导出；
   * 发送事件：`bus.emit('事件名', 数据)`；
   * 接收事件：`bus.on('事件名', 回调函数)`；
   * 移除事件：`bus.off('事件名', 回调函数)`。
2. **避免内存泄漏的方法**：
   * 组件卸载时（`onUnmounted` 钩子），调用 `bus.off` 销毁对应的事件监听；
   * 若需移除所有监听，可调用 `bus.all.clear()`；
   * 核心逻辑：组件卸载后不再监听事件，防止无用的回调函数占用内存。

#### 题目 5：自定义 v-model 如何实现？Vue3 中 v-model 和 Vue2 有什么区别？

**标准答案**：

1. **Vue3 自定义 v-model 实现**：
   * 子组件：`defineProps(['xxx'])` 接收属性，`defineEmits(['update:xxx'])` 声明事件，修改数据时触发 `emit('update:xxx', 新值)`；
   * 父组件：`<Child v-model:xxx="value" />`。
2. **Vue3 vs Vue2 v-model 区别**：
   * Vue2：默认绑定 `value` 属性 + `input` 事件，一个组件只能有一个 v-model，自定义需修改 `model` 选项（`model: { prop: 'xxx', event: 'change' }`）；
   * Vue3：取消默认 `value/input` 绑定，支持**多个 v-model**（`v-model:name`/`v-model:age`），语法更灵活，无需修改 model 选项，直接通过 `update:属性名` 事件实现。

#### 题目 6：Vuex 和 Pinia 的区别？为什么 Vue3 推荐使用 Pinia？

**标准答案**：

1. **核心区别**：
   表格


   | 维度       | Vuex                            | Pinia                                     |
   | ---------- | ------------------------------- | ----------------------------------------- |
   | 模块化     | 需通过 Modules 实现             | 天然模块化，每个 Store 独立               |
   | Mutations  | 必须通过 Mutations 修改 State   | 无 Mutations，直接在 Actions 中修改 State |
   | TS 支持    | 需手动注解，体验差              | 天然适配 TS，类型推导完善                 |
   | 嵌套结构   | 支持嵌套 Modules                | 扁平结构，无嵌套                          |
   | 代码简洁性 | 繁琐（State/Mutations/Actions） | 简洁，仅 State/Actions/Getters            |
   | 调试工具   | 支持                            | 支持，且调试体验更好                      |
2. **Vue3 推荐 Pinia 的原因**：

   * 更适配 Vue3 的 Composition API，支持解构和 TS；
   * 简化代码，去掉冗余的 Mutations，开发效率更高；
   * 天然模块化，无需嵌套，代码结构更清晰；
   * 体积更小，性能更优。

#### 题目 7：ref/\$refs 和 provide/inject 的区别？分别适用于什么场景？

**标准答案**：

1. **核心区别**：
   * ref/\$refs：父组件直接获取子组件实例，可访问子组件的属性 / 方法，**仅支持父子组件**，耦合度高；
   * provide/inject：父组件提供数据，任意后代组件注入使用，**支持跨层级**，耦合度低（但数据来源不清晰）。
2. **适用场景**：
   * ref/\$refs：父组件需主动调用子组件方法（如表单组件的重置方法）、获取子组件的 DOM 元素；
   * provide/inject：深层嵌套组件传递公共数据（如主题配置、用户信息），无需层层传递 props。

#### 题目 10：provide/inject 如何实现响应式？手写示例

**标准答案**：

provide/inject 天然支持响应式，只需提供**响应式数据**（ref/reactive），后代组件注入后会同步更新：

vue

```
<!-- 父组件 -->
<script setup>
import { provide, reactive, readonly } from 'vue'
// 提供响应式数据（readonly 防止后代修改）
const appConfig = reactive({
  theme: 'light',
  size: 'medium'
})
provide('appConfig', readonly(appConfig))

// 提供修改方法
provide('updateTheme', (newTheme) => {
  appConfig.theme = newTheme
})
</script>

<!-- 后代组件 -->
<script setup>
import { inject } from 'vue'
// 注入响应式数据
const appConfig = inject('appConfig')
const updateTheme = inject('updateTheme')

// 修改数据（通过父组件提供的方法）
const changeTheme = () => {
  updateTheme('dark')
}
</script>
<template>
  <div>当前主题：{{ appConfig.theme }}</div>
  <button @click="changeTheme">切换暗黑模式</button>
</template>
```

### 拓展追问（大厂高频）

#### 题目 11：为什么不推荐用 \$parent / \$children 通信？

**标准答案**：

1. 依赖组件层级：若组件结构调整（如新增 / 删除嵌套层级），`$parent/$children` 指向会变化，导致代码报错；
2. 耦合度高：子组件直接依赖父组件的结构 / 方法，父组件修改后子组件需同步修改；
3. 可读性差：无法直观看出数据 / 方法的来源，维护成本高；
4. Vue3 弱化：Vue3 中 `$children` 已不再推荐使用，建议用 ref/provide-inject 替代。

#### 题目 12：Pinia 的核心原理？和 Vuex 相比有什么优势？

**标准答案**：

1. **Pinia 核心原理**：
   基于 Vue3 的响应式系统（Proxy）和 Composition API 实现，每个 Store 是一个独立的响应式对象，通过 `defineStore` 创建，支持按需导入，无需像 Vuex 一样挂载到 Vue 实例。
2. **相对 Vuex 的优势**：
   * 无 Mutations：直接在 Actions 中修改 State，简化代码；
   * 天然 TS 支持：类型推导完善，无需手动注解；
   * 模块化更简单：每个 Store 独立，无需嵌套 Modules；
   * 体积更小（约 1KB），性能更优；
   * 支持解构：可直接解构 Store 中的数据（配合 toRefs），无需 mapXXX 辅助函数。

#### 题目 13：组件通信中如何保证单向数据流？

**标准答案**：

单向数据流是 Vue 的核心设计原则，保证方式如下：

1. 子组件不直接修改 props，需通过 `emit` 通知父组件修改；
2. provide/inject 提供的数据用 `readonly` 包装，禁止后代直接修改，仅通过父组件提供的方法修改；
3. 全局状态（Vuex/Pinia）通过统一的方式修改（Vuex 的 Mutations/Actions，Pinia 的 Actions），避免任意组件直接修改；
4. 避免使用 ref/\$refs 直接修改子组件的状态，仅在必要时调用子组件的方法。

#### 题目 14：事件总线和 Pinia 如何选择？

**标准答案**：

核心按「项目规模」和「数据类型」选择：

1. **选事件总线（mitt）**：
   * 中小型项目，临时 / 简单的跨组件通信（如通知弹窗显示 / 隐藏）；
   * 无需持久化、无需追踪的一次性事件；
2. **选 Pinia**：
   * 大型项目，需全局共享的状态（如用户信息、购物车、主题配置）；
   * 需追踪状态变化、支持模块化、适配 TS 的场景；
   * 需持久化状态（如 localStorage 缓存）的场景。

#### 题目 15：Vue3 中 \$attrs 包含哪些内容？如何透传？

**标准答案**：

1. \*\*listeners`已合并到`\$attrs`），如 `class`/`style`/`id\`、自定义事件等。
2. **透传方式**：
   子组件通过 `v-bind="$attrs"` 将属性 / 事件透传给内部的 DOM 或子组件：vue

   ```
   <!-- 子组件 -->
   <template>
     <!-- 透传所有 attrs 到 input 标签 -->
     <input v-bind="$attrs" />
   </template>
   <script setup>
   // 声明接收的 props，未声明的会进入 $attrs
   defineProps(['value'])
   </script>
   ```

## 四、核心总结

1. **通信方式选择核心**：父子用 Props/\$emit/v-model，跨层级用 provide/inject，任意组件中小型项目用 mitt、大型项目用 Pinia；
2. **面试答题关键**：先按层级分类说明通信方式，再结合场景讲优缺点，最后手写核心示例（如自定义 v-model、mitt、provide/inject）；
3. **避坑点**：props 不能直接修改、事件总线需销毁监听、provide/inject 配合 readonly 使用，保证单向数据流。
